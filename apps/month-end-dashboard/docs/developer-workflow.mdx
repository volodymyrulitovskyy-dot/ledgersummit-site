## 2 — Environment Variables

### Quick path (recommended)

If you have access to the sibling `month-end-dashboard-2` repo, run the helper script:

```bash
./scripts/copy-env.sh ../month-end-dashboard-2/.env.local
```

### Manual path

```bash
cp .env.example .env
```

Then fill in your values:

| Variable | Required | Description |
|----------|----------|-------------|
| `DATABASE_URL` | ✅ | Supabase Postgres connection string. **Do NOT append `?schema=med3`** — Prisma handles schema isolation via `@@schema("med3")` in the schema file. |
| `NEXT_PUBLIC_SUPABASE_URL` | ✅ | Your Supabase project URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | ✅ | Supabase anonymous/public key |
| `SUPABASE_SERVICE_ROLE_KEY` | optional | Service role key (server-side admin ops) |
| `NEXT_PUBLIC_APP_URL` | optional | Defaults to `http://localhost:3015` |
| `ADMIN_EMAIL` | optional | Email address that is auto-promoted to admin on first login |

### QuickBooks Online (optional)

| Variable | Description |
|----------|-------------|
| `QBO_CLIENT_ID` | Intuit OAuth app client ID |
| `QBO_CLIENT_SECRET` | Intuit OAuth app client secret |
| `QBO_REDIRECT_URI` | OAuth callback — default `http://localhost:3015/api/qbo/callback` |
| `QBO_ENV` | `sandbox` or `production` |

You can copy QBO vars from the sibling project automatically:

```bash
./scripts/copy-qbo-env.sh ../month-end-dashboard-2 .env.local
```

<Callout type="warning">
  **Never commit `.env` files.** They are git-ignored. Only `.env.example` is tracked.
</Callout>

## 3 — Database Setup

### 3.1 — Create schemas in Supabase

Before any Prisma commands, run the schema-creation SQL in the **Supabase SQL Editor**:

```sql
-- Located at: scripts/create_schemas.sql
-- Creates med3 and med3_shadow schemas and grants permissions
```

Then expose the schema:

1. Go to **Settings → API** in the Supabase dashboard.
2. Under **Exposed schemas**, add `med3` to the list.

### 3.2 — Run Prisma migrations

**First-time setup:**

```bash
npx prisma migrate dev --name init
```

If the shadow-database step fails (common with hosted Supabase):

```bash
npx prisma migrate dev --create-only --name init
npx prisma migrate deploy
npx prisma generate
```

**Subsequent deploys (migrations already exist):**

```bash
npx prisma migrate deploy
npx prisma generate
```

### Key patterns

- **App Router** — All routing via `src/app/`. Route groups `(app)` (protected) and `(public)` control auth boundaries.
- **Server Actions** — Heavy use of Next.js server actions (`"use server"`) for data mutations.  
  Look for `actions.ts` files inside each feature folder.
- **Prisma multi-schema** — All models live in the `med3` Postgres schema, declared via `@@schema("med3")` in `schema.prisma`.
- **Supabase Auth** — Email/password authentication. Middleware redirects unauthenticated users to `/auth/login`.
- **RBAC** — Two roles: `admin` and `user`. Screen access is enforced server-side. Admin manages users at `/admin/users`.
- **React Compiler** — Enabled in `next.config.ts` (`reactCompiler: true`).

## 6 — Authentication & Authorization

### Auth flow

1. User hits any `(app)` route → `middleware.ts` checks Supabase session.
2. No session → redirect to `/auth/login`.
3. After login → `requireUser()` in server components fetches user + role + allowed screens.
4. Unauthorized screen access → redirect to `/access-denied`.

### Roles

| Role | Access |
|------|--------|
| `user` | Home, Checklist, Calendar (default for new signups) |
| `admin` | All screens + Admin panel (`/admin/users`) |

### Smoke checks

- **User** logs in → sees only Home / Checklist / Calendar in nav.
- **User** blocked from `/rules` or `/admin/users` → redirects to access denied.
- **Admin** can access `/rules` and `/admin/users`.
- **Exceptions** are scoped: user sees own (`owner_user_id`), admin sees all.

```bash
# 1. Edit prisma/schema.prisma (remember @@schema("med3"))
# 2. Create migration
npx prisma migrate dev --name add_my_table

# 3. Regenerate client
npx prisma generate
```

### Browse the database visually

```bash
npm run db:studio
```

Opens **Prisma Studio** at `http://localhost:5555`.

### Seed data

```bash
# Close tasks
npx tsx scripts/seed-close-tasks.ts

# Trial balance
npx tsx scripts/seed-trial-balance.ts
```

### Add a new protected page

1. Create a folder under `src/app/(app)/your-feature/`.
2. Add `page.tsx` with `requireUser()` at the top.
3. Add the route to the nav in `src/app/(app)/AppNav.tsx`.
4. Update RBAC allowed-screens if needed (admin panel manages this).

### Add a new API route

1. Create `src/app/api/your-endpoint/route.ts`.
2. Export named functions: `GET`, `POST`, `PUT`, `DELETE`, etc.
3. Use `requireUser()` for auth-gated endpoints.

## 10 — Troubleshooting

### Prisma migration fails with permission errors

Ensure you've run `scripts/create_schemas.sql` in Supabase SQL Editor first. The `med3` and `med3_shadow` schemas must exist with proper grants.

### Shadow database errors

Use the two-step migration approach:

```bash
npx prisma migrate dev --create-only --name my_migration
npx prisma migrate deploy
npx prisma generate
```

### `Decimal` / `toFixed` errors in client components

Prisma returns `Decimal` objects. Convert to plain numbers **before** passing props to client components:

```typescript
const value = Number(prismaRow.amount);
```

### Tables not visible via Supabase client

Make sure `med3` is listed under **Settings → API → Exposed schemas** in your Supabase dashboard.

### Port conflicts

The dev server runs on port **3015** (configured in `package.json` → `"dev": "next dev -p 3015"`).  
Change the port there if needed.
